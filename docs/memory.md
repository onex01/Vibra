# Подсистема памяти Vibra OS

Документ описывает, как ядро управляет памятью на этапе «Фундамент + память»
(ветка `main`, OS 0.5 «Nucleus»). Здесь разбираются PMM (физическая память) и
heap-аллокатор (куча для `Vec`/`String`/`Box`) и начальный read-only слой
виртуальной памяти. Собственные таблицы страниц ещё не переключаются: на
первом подшаге Шага 4 ядро читает CR3 и проверяет таблицы Limine.

> Связанные файлы: `src/memory/pmm.rs`, `src/memory/heap.rs`, `src/memory/mod.rs`,
> `src/main.rs` (порядок init).

---

## 1. Обзор

Ядро работает в long mode, таблицы страниц строит загрузчик **Limine**, и до
этапа своих page tables мы не трогаем адресное пространство напрямую. Тем не
менее для `Vec`/`String` нужен работающий аллокатор, а для него — учёт
физической памяти. Поэтому подсистема памяти сейчас состоит из трёх частей:

| Слой | Назначение | Что делает |
|------|-----------|------------|
| **PMM** (`pmm.rs`) | Физическая память | Битмап фреймов по 4 КБ, выделение/освобождение |
| **Heap** (`heap.rs`) | Куча для `alloc` | Free-list с коалесценцией поверх региона из PMM |
| **Paging** (`paging.rs`) | Диагностика виртуальной памяти | Читает CR3 и переводит virt → phys через PML4/PDPT/PD/PT |

Между ними — **HHDM** (Higher Half Direct Map): Limine маппит всю физическую
память в фиксированный виртуальный регион, поэтому PMM-фрейм превращается в
доступный указатель одной операцией `virt = hhdm + phys`. Это позволяет heap'у
и (в будущем) page tables работать с физической памятью без собственной
настройки отображения.

```
   PMM (физические фреймы)
        │  alloc_contiguous(1024) ─► phys
        ▼
   heap.rs:  virt = hhdm + phys  ──► FreeList на 4 МБ
        │
        ▼
   #[global_allocator]  ──► Vec / String / Box
```

Чего здесь **нет** (задел следующих этапов):
- собственных page tables / собственного адресного пространства ядра (следующий
  подшаг Шага 4);
- защиты памяти и ring 3 (этап планировщика/user mode);
- подкачки (swap) и файлового кэша.

## 2. HHDM — Higher Half Direct Map

Limine по запросу `HhdmRequest` маппит **всю** физическую память в один
непрерывный виртуальный регион с фиксированным смещением `hhdm_offset`
(запрашивается в `main.rs` через `HHDM_REQUEST`). Любой физический адрес
`phys` доступен по виртуальному адресу

```
virt = hhdm_offset + phys
```

отображение 1:1, `PRESENT | WRITABLE | NX`, страницами 2 МБ.

Это удобство мы используем в двух местах:

1. **Heap** берёт 4 МБ у PMM и работает с ними как с виртуальным массивом
   байт — потому что HHDM уже всё отобразил.
2. **`pmm::alloc_frame_zeroed(hhdm)`** (задел под Шаг 4) выделяет фрейм и
   возвращает его HHDM-адрес, чтобы page-tables могли писать в таблицы
   напрямую.

Важно: HHDM намеренно покрывает и `BOOTLOADER_RECLAIMABLE`-регионы (там живёт
стек BSP, который Limine выделил при загрузке), и framebuffer. Поэтому PMM не
должен отдавать reclaimable-память как свободную — иначе можно раздать
собственный стек (см. §3).

## 3. PMM — Physical Memory Manager

Файл: `src/memory/pmm.rs`.

### Принцип: битмап

Один бит на 4 КБ-фрейм: `0` — свободен, `1` — занят. Размер битмапа
`BITMAP_SIZE = MAX_FRAMES / 8`, где `MAX_FRAMES = 1_048_576` (хватает на 4 ГБ).
Битмап живёт в `.bss` и инициализируется константой `[0xFF; ...]` —
«всё занято». `init()` размечает свободными только `MEMMAP_USABLE`-регионы
из карты памяти Limine.

```
битмап:  [1][1][0][0][0][1][0][0] ...   (бит = фрейм)
фрейм:    0  1  2  3  4  5  6  7
              └── usable-регион ──┘     └ раздаём отсюда
```

### Почему НЕ reclaimable

Limine помечает часть памяти как `BOOTLOADER_RECLAIMABLE` — структуры самого
загрузчика, **и в том числе стек BSP**, на котором крутится `_start` до
появления своего стека. PMM намеренно не помечает reclaimable как свободный:
раздать собственный стек = почти гарантированный краш при первом же прерывании.
Эта память будет освобождена на этапе планировщика, когда появится свой
per-thread стек и ядро уйдёт с Limine-стека.

### Read-only paging walker

`memory::paging` сохраняет HHDM offset, читает CR3 и маскирует PCID-биты,
получая физический адрес активной PML4. Затем `translate()` обходит уровни
PML4 → PDPT → PD → PT и учитывает 4 КиБ, 2 МиБ и 1 ГиБ huge pages. Он только
читает таблицы через HHDM и не меняет CR3, поэтому безопасен как первый шаг
перед построением собственного адресного пространства. В shell команда
`diag paging` проверяет отображение `_start`.

`diag paging-test` идёт на шаг дальше: выделяет временные PML4, PDPT, PD, PT
и физическую страницу, копирует текущую PML4, размещает страницу в свободном
PML4-slot и проверяет её через `translate_from_root`. CR3 при этом не меняется,
а пять фреймов после проверки возвращаются PMM.

### Состояние и локи

Раньше PMM держал состояние в `static mut` (`BITMAP`, `FREE_FRAMES`). Сейчас
это:

```rust
static BITMAP: Mutex<[u8; BITMAP_SIZE]> = Mutex::new([0xFF; BITMAP_SIZE]);
struct PmmInner { free_frames, total_frames, next_hint }
static PMM: Mutex<PmmInner> = Mutex::new(PmmInner::new());
```

Все операции берут лок **внутри `interrupts::without_interrupts()`** — это
страховка от дедлока под будущий вытесняющий планировщик: ISR никогда не
отберёт лок у держателя, потому что прерывания выключены на время операции.
На старте прерывания и так выключены, но единообразие локов важно.

### Next-fit вместо линейного скана

`alloc_frame()` раньше сканировал битмап с нуля на каждый вызов — это O(N) в
худшем случае и порт локальность. Теперь работает **next-fit**: поиск
начинается с `next_hint`, при успехе `next_hint` сдвигается за найденный фрейм,
при достижении конца — wrap на 0. Это держит «активную зону» компактной и
быстро находит свободные фреймы при разреженном использовании.

### Публичный API

| Функция | Возвращает | Назначение |
|---------|-----------|------------|
| `init(memory_map)` | `()` | Разметить USABLE-регионы, посчитать счётчики |
| `alloc_frame()` | `Option<usize>` (физ. адрес) | Один 4 КБ-фрейм (next-fit) |
| `free_frame(addr)` | `()` | Освободить фрейм (с детектом double-free) |
| `alloc_contiguous(count)` | `Option<usize>` | `count` подряд идущих фреймов (для heap) |
| `alloc_frame_zeroed(hhdm)` | `Option<u64>` (вирт. адрес) | Чистый фрейм через HHDM (задел под page tables) |
| `stats()` | `(used, total)` | Счётчики фреймов |

`alloc_contiguous` нужен heap'у (4 МБ = 1024 фрейма). `alloc_frame_zeroed`
пока не вызывается (`#[allow(dead_code)]`) — он для Шага 4: новым page tables
нужны гарантированно чистые страницы.

## 4. Heap — free-list аллокатор

Файл: `src/memory/heap.rs`.

### Бэкенд: PMM + HHDM, а не статика

Старый `BumpAllocator` жил в 1 МБ-статике `HEAP_MEMORY` в `.bss`. Это работало,
но не масштабировалось и не освобождало память. Новый аллокатор берёт регион
**у PMM** и обращается к нему через **HHDM**:

```rust
// heap::init(hhdm)
let phys = pmm::alloc_contiguous(1024)?;   // 4 МБ физикой
let start = hhdm + phys;                    // виртуальный адрес
FreeList::init(start, 4 * 1024 * 1024);     // работаем как с массивом байт
```

Это убирает 1 МБ из `.bss` и делает кучу настоящей — с `free` и
коалесценцией. Размер 4 МБ выбран с запасом под heap-стресс (10k итераций) и
будущий рост; при нехватке 4 МБ аллокатор пробует 2 МБ, иначе паникует.

### Структура свободного блока

Узел списка **overlay-ится поверх самого блока** — никаких отдельных
дескрипторов:

```
свободный блок:
 ┌──────────────────────────────────────────┐
 │ size: usize   │  next: Option<NonNull>   │  ← FreeNode (MIN_BLOCK = 32 Б)
 ├──────────────────────────────────────────┤
 │                ...пусто...                │
 └──────────────────────────────────────────┘
```

Поэтому минимальный размер блока = `size_of::<FreeNode>()` с округлением до 16
(на x86_64 — 32 байта). Любой блок меньше `MIN_BLOCK` не может стать узлом и
остаётся «прилипшим» к соседу (см. ниже).

### Address-ordered singly-linked list

Список свободных блоков **отсортирован по адресу**. Это ключ ко всему:

- **`allocate`** — first-fit: идём по списку, берём первый блок, вмещающий
  выровненный регион. Делим: пользователь получает начало, остаток справа
  возвращается в список новым узлом (если остаток ≥ `MIN_BLOCK`). Если остаток
  слишком мал — отдаём весь блок целиком (лишние байты «прилипают»).
- **`deallocate`** — вставляем блок в нужное место по адресу и
  **коалесцируем** с обоими соседями: если левый сосед кончается там, где
  начинается освобождаемый блок — сливаем; то же с правым. Так список не
  фрагментируется.

Все размеры округляются вверх до 16-байтной гранулы (`ALIGN_GRANULE`).
`Layout::size() == 0` возвращает `dangling()` по контракту `GlobalAlloc`.

### Локи и OOM

Доступ сериализован через `Mutex<FreeList>`, причём `alloc`/`dealloc` берут
лок внутри `without_interrupts()` — как и PMM. На OOM (нет блока подходящего
размера) аллокатор печатает в serial размер запроса и текущие `stats`, и
возвращает `null_mut()` — вызывающий `Vec`/`Box` упадёт с паникой роста.

`LockedHeap` помечен `unsafe impl Send + Sync`: `NonNull` сам по себе не
`Send`, но однопоточное ядро + доступ строго под локом делают совместное
использование безопасным.

### Форма API

Форма совместима с `linked_list_allocator::Heap` (`init(start, size)`), поэтому
если отладка собственного аллокатора затянется, крейт вставляется без правки
вызывающего кода. Пока — пишем сами (~250 строк) ради педагогической ценности.

## 5. Порядок инициализации

Из `main.rs::_start()` и `memory/mod.rs`:

| Шаг | Что | Аллоцирует heap? |
|-----|-----|:---:|
| 1 | `serial::init()` | — |
| 2 | `pmm::init(memory_map)` — битмап, счётчики | — |
| 3 | `heap::init(hhdm)` — `alloc_contiguous(1024)`, FreeList | — (она сама создаёт кучу) |
| 4 | **heap-стресс** — 10k alloc/drop, проверка `used → 0` | да (временно, всё освобождается) |
| 5 | `keyboard::init()` | — |
| 6 | `fs::init_filesystem()` — RamFS на статических массивах | — |
| 7 | `kernel::init()` → `module::register_builtin` ×3 | **да** (постоянно: `String`/`Vec`) |
| 8 | `device::register` ×5, `driver::register` ×3 | **да** (постоянно) |
| 9 | `gdt::init()`, `interrupts::init()`, `interrupts::enable()` | — |

Ключевой момент: **heap обязан быть готов до `kernel::init()`** (шаг 7) — там
первые постоянные аллокации (`String::from`, `Vec::push`). Поэтому
`heap::init()` вызывается внутри `memory::init()` сразу после `pmm::init()`.

## 6. Тесты

### `diag pmtest` — тест PMM (Шаг 2)
Shell-команда. Делает: `stats` до → `alloc_contiguous(16)` → 1000×
`alloc_frame` → free всех → `stats` после. Ожидаемый результат: **PASS** —
счётчики `used` в начале и конце совпадают (доказывает корректность
`free` + next-fit + contiguous).

### Heap-стресс в `main.rs` (Шаг 3)
Запускается при загрузке, **до** `kernel::init()`. 10 000 итераций: на каждой
`Vec` размером 8..255 байт, заполненный проверяемым паттерном, проверка
целостности, `drop`. Раз в ~2000 итераций держит несколько живых блоков
одновременно для реальной фрагментации. Ожидаемый serial-вывод:

```
[HEAP] stress: 10000 iters OK (used 0 -> 0)
```

Если `used` разошёлся — `[HEAP] stress: 10000 iters LEAK <N>` (free или
коалесценция сломаны). Коррупция данных ловится как
`[HEAP] stress: CORRUPTION at iter <i>`.

### `heap` — shell-команда
Показывает текущее `stats()`: used / free / total в байтах и КБ, процент
заполнения. Полезно после стресс-теста и в рантайме для отладки утечек.

## 7. Задел на будущее

Реализовано сейчас, но ждёт следующих этапов:

- **`pmm::alloc_frame_zeroed(hhdm)`** — готов к использованию в Шаге 4
  (свои page tables): новые таблицы нуждаются в чистых страницах.
- **GDT-слоты `0x18`/`0x20`** зарезервированы под user data/code — порядок
  подобран под будущий `sysret` (ring 3).
- **TSS.rsp0** — структура уже несёт массив `rsp[3]`; понадобится для ring 3.
- **`MIN_BLOCK`/коалесценция** — задел под долгоживущую фрагментированную
  кучу, когда появится планировщик и постоянные аллокации процессов.

Этап планировщика (вытесняющий, kernel-threads) неявно требует: уйти с
Limine-стека на per-thread стеки — а для этого нужно освободить
`BOOTLOADER_RECLAIMABLE`, что сейчас намеренно запрещено.
